# Tech Insight Synthesizer

## Overview
這個 Skill 是用來寫「知識分享文」的，不是技術文件、不是架構規格書、也不是給 CTO 看的內部報告。

判斷標準很簡單：寫出來的東西，如果拿給一個對你們產品好奇、但完全不懂技術的朋友看，他願意讀完嗎？如果答案是「他會看到第二段就放棄」，那這篇就寫錯了。

**常見的失敗示範（不要這樣）：**
> 當 Sim2Real 不再只是「把模型丟進真機」：Robotic Suite 如何把校正、資料覆蓋與長流程模擬收斂成可操作平台

這個標題的問題不是內容不對，是它讀起來像論文摘要——一句話塞了四五個專有名詞，句子長到要喘氣，完全沒有勾人的感覺。

**改成這樣才對：**
> 當模擬不再等於現實：Robotic Suite 如何跨越機器人落地最大的鴻溝

差別在哪？短、有反差感、用「鴻溝」這種生活詞取代術語堆疊，讀者一秒就懂這篇在講什麼問題，但又好奇答案是什麼。

**核心原則：**
- **先讓人好奇，再講技術**：開頭丟一個情境、一個反差、或一句讓人意外的話，不要丟摘要、不要丟條列重點
- **人話優先**：每個專有名詞出現前，先想「這個東西如果要跟外婆解釋，我會怎麼說」，背後的事實和數字不能因此走味，但語言要換
- **只寫真的**：圖片、程式碼、數字都要來自專案實際留存的素材，找不到依據就老實說「目前還不確定」，不可以編
- **有真實圖片就用**：`_assets/` 裡真的相關的截圖、架構圖，比文字描述更有說服力

若由 `project-insight-synthesizer` 觸發，本 Skill 產出的內容會整併進既有專案 Markdown；獨立呼叫時則輸出一篇獨立文章。

## 輸入
- **產品／技術方案名稱**（必要）：文章主角
- **主題**（選擇性）：例如架構設計、效能優化、AI 整合、安全機制；沒指定就自己從資料裡挑「最有看頭」的主題
- **專案 Markdown 或會議摘要**（選擇性）：由 project-insight-synthesizer 傳入，是事實的來源
- **圖片素材**（選擇性）：沒有直接傳入的話，自己去 `project-insights/record-<slug>/` 和 `_assets/` 找真的相關的截圖
- **目標讀者**（選擇性）：沒指定就預設「對這個技術有點好奇但沒背景」的一般人；只有使用者明確說要寫給工程師／架構師看時，才往技術細節傾斜
- **輸出目標**（選擇性）：`standalone`（獨立文章）或 `merge`（整併進既有專案 Markdown）

## 怎麼動筆

### 1. 找出這篇文章真正的「亮點」
從專案資料（會議摘要、逐字稿、專案 Markdown）裡找事實，標好來源日期，例如 `來源：2026-04-15 專案記錄`。這一步不要混入任何外部推論——先搞清楚「我們真的做了什麼」，再決定怎麼講。

找的時候問自己：這件事在業界少見嗎？難在哪？讀者看完能學到什麼？有沒有具體數字或案例佐證？選 1-3 個最值得寫的點，不要想著面面俱到——什麼都講等於什麼都沒講清楚。

### 2. 把真實圖片找出來，親眼看過
1. 用 Glob／ls 看 `project-insights/_assets/` 實際有哪些檔案——這是文章裡唯一能引用的圖片來源，檔名要完全對上
2. 讀 `project-insights/record-<slug>/*.md`，裡面通常已經有 `![](../_assets/<file>)` 的圖片連結，可以先用這些縮小範圍
3. 每張候選圖片都要用 Read 工具親眼打開看內容，確認它真的支持你要講的點，不能只憑檔名或附近文字猜
4. 留下真的有用的圖，記下檔名、實際看到什麼、要放哪一段；找不到合適圖片的段落就用文字寫，不要硬塞一張不相關的圖,也不能编一個不存在的檔名

### 3. 查一點背景資料，讓讀者覺得「這不是我們自己的怪問題」
搜尋外部資料的目的不是寫一個比較表，是讓故事更可信。可以查：官方文件與規範、開源專案的做法、知名工程部落格（Netflix、Uber、Cloudflare 那種）、技術社群討論、相關論文。

找不到足夠可信的資料就老實說「外部技術佐證不足」，不要用猜測填補。每條外部資料都要記 `外部技術資料：<標題> — <URL>（<日期>）`，之後要自然地揉進故事裡講，不要另開一個「業界對照表」章節——一句「Netflix 在 2023 年也遇過類似狀況」帶過去就好。

### 4. 動筆寫
寫之前先想：這篇故事怎麼鋪陳最自然？不用照搬下面的順序，但通常會包含這些感覺：

- 一開場就讓人好奇——用情境、提問或一個意外的事實，不要用摘要或專有名詞開場
- 讓讀者覺得這問題很普遍——把外部資料自然帶進這段
- 我們真正卡住的地方——具體說困難在哪，用生活化的類比
- 怎麼想通的——關鍵的轉折跟決策，類比先講，技術細節再補
- 真正做出來時發生了什麼——故事化地講過程，有真實截圖就嵌進去
- 後來呢——讀者有感的結果（「以前要等三天，現在十分鐘」這種講法），別寫成效能報表
- 踩過的坑——通常是整篇最真實、最好看的部分，老實講
- 如果你也遇到類似問題——讀者能怎麼用上這些經驗
- 想多看一點——給有興趣的人的延伸連結

標題和每個小節的文字，都要像「人在說話」，不要像投影片標題或論文摘要。寧可用一個短句加一點懸念，也不要把名詞全部塞進一句話。

**程式碼規範（嚴格）**：不寫 pseudo code，不捏造程式碼片段。只能逐字引用專案資料、逐字稿、會議記錄裡真實出現過的程式碼或設定，並標來源。沒有真實程式碼就用文字描述邏輯，不要假裝有程式碼範例。

### 5. 出手前再檢查一次
- 文章裡每個 `![...](../_assets/<file>)` 對應的檔案，都要在第 2 步列出的真實清單裡，對不上的整段拿掉，不能留 404 連結
- 沒有真實依據的程式碼、數字、圖說，都已經拿掉或改成「目前還不確定」
- 把整篇文章從頭讀一次：有沒有哪一段聽起來像在唸規格書？有就重寫那一段

## 輸出

`standalone`：輸出至 `project-insights/techblog-<project-slug>.md`，內容大致包含：
- 一個勾人但不誇張的標題（短句、有反差或懸念，不堆專有名詞）
- 作者視角、發表日期、適合讀者、預計閱讀時間
- 用故事鋪陳的正文（見上方「怎麼動筆」第 4 步的元素，自然分段，不用硬套固定標題）
- 真實圖片直接嵌入，標來源；真實程式碼逐字引用，標來源
- 文末附資料透明度說明（技術事實來源、外部資料完整性高/中/低、主要不確定性）和來源索引

`merge`（由 project-insight-synthesizer 觸發）：回傳結構化內容，包含技術敘事、有嵌入的圖片清單（檔名/圖說/來源）、效益敘事、揉入故事的外部對照觀察、完整來源列表、信心程度（high/medium/low）、資料缺口說明。

## 標記來源的方式
- 專案事實：`來源：YYYY-MM-DD 專案記錄` 或 `來源：YYYY-MM-DD 逐字稿`，只能寫專案資料裡真的出現過的東西
- 外部資料：`外部技術資料：<標題> — <URL>（<日期>）`，任何不是來自專案本身的技術主張都要附這個，不能偽裝成原創發現
- 圖片：`![圖說](../_assets/<file>)` 緊接 `來源：YYYY-MM-DD 專案記錄`，圖說要寫你實際在圖裡看到的東西，不能套用檔名或憑空想像
- 資料不足時：老實寫「外部技術佐證不足，以下為內部觀察，尚待與業界做法比對」，不用猜測填補

## 回報給使用者
完成後回報：
1. 文章主題與目標讀者
2. 核心亮點（2-3 點）
3. 外部資料完整性，哪些主張有外部佐證、哪些純來自專案事實
4. 嵌入了哪些真實圖片（檔名與來源），哪些段落改用文字描述
5. 輸出檔案位置（standalone）或整併狀態（merge）
6. 建議的下一個知識分享題材

## Integration with project-insight-synthesizer
若由 `project-insight-synthesizer` 觸發：
1. 對方傳入：專案名稱、專案 Markdown 摘要、技術主題、輸出目標 `merge`
2. 本 Skill 完成草稿後回傳結構化內容
3. project-insight-synthesizer 整併進專案 Markdown 的「技術亮點與對外發表素材」章節，保留來源標記
