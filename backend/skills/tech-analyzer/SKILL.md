# Tech Insight Synthesizer

## Overview
從 CTO 與資深工程師的視角出發，將專案的核心技術架構、設計決策與突破點，整理成一篇可對外發表的技術文章。

**核心原則：**
- 聚焦「為什麼這麼做」的決策脈絡，而非只羅列「做了什麼」
- 技術判斷必須附上具體依據（指標、架構圖說明、對比數字）
- 外部技術資料用於佐證，不取代專案自身的事實
- 文章要讓讀者學到東西，不只是廣告
- 嚴格區分專案事實與外部技術研究

若由 `project-insight-synthesizer` 觸發，本 Skill 產出的技術文章區塊將整併進既有專案 Markdown；若為獨立呼叫，輸出獨立的技術部落格文章。

## 輸入

| 欄位 | 必要性 | 說明 |
|------|--------|------|
| 產品／技術方案名稱 | 必要 | 文章主角 |
| 技術方向或主題 | 選擇性 | 例如：架構設計、效能優化、AI 整合、安全機制；未指定則由 Skill 從資料中判斷最有看點的主題 |
| 專案 Markdown 或會議摘要 | 選擇性 | 由 project-insight-synthesizer 傳入；作為技術事實來源 |
| 目標讀者 | 選擇性 | 例如：後端工程師、DevOps、架構師、CTO；未指定則預設為有技術背景的工程師 |
| 文章深度 | 選擇性 | `intro`（概念介紹，適合入門）/ `deep-dive`（深度技術，預設）/ `decision-log`（架構決策記錄） |
| 輸出目標 | 選擇性 | `standalone`（獨立技術文章）或 `merge`（整併進既有專案 Markdown） |

## Workflow

### Step 1: 確認文章主題與深度
依使用者輸入，確認這篇文章要說什麼、給誰看：

**文章類型選擇：**
- **架構分析**：說明整體技術架構的設計理念與取捨
- **問題解決**：聚焦某個具體技術挑戰，說明如何解決
- **效能優化**：展示具體的優化前後對比與方法
- **技術選型**：說明為什麼選這個技術棧而非其他選項
- **實作心得**：分享踩坑經驗與最佳實踐

若未指定，從傳入資料中判斷「最有說服力的技術突破點」作為主題。

### Step 2: 提取並隔離技術事實
若有傳入專案 Markdown 或會議摘要，先提取技術事實，標記來源日期：
- 技術架構的具體描述
- 設計決策與取捨說明
- 具體的效能指標或測試結果
- 遇到的技術挑戰與解法
- 已驗證的技術突破（PoC、壓測、上線數據）
- 未來技術路線圖（若有）

**這些是文章的第一手技術事實，標記為：`來源：2026-04-15 專案記錄`**
**不可在此步驟混入外部推論或外部技術資料。**

### Step 3: 識別核心技術突破點
從事實中識別「最值得寫的技術亮點」：
- **創新性**：這個做法在業界不常見嗎？
- **挑戰性**：解決了什麼難題？難在哪裡？
- **可學習性**：讀者看完能學到什麼？
- **可驗證性**：有具體數字或案例佐證嗎？

選出 1-3 個核心突破點作為文章主軸，避免面面俱到但深度不足。

### Step 4: 執行外部技術研究
針對文章主題，搜尋外部技術資料以：
1. 建立背景知識（讓讀者理解問題的普遍性）
2. 與業界主流做法做對比
3. 補充相關技術標準或規範
4. 引用社群驗證（GitHub、論文、技術社群討論）

搜尋優先順序：
1. 官方技術文件與規範（RFC、W3C、IETF）
2. 開源專案 repo 與官方 docs（GitHub、npm、PyPI）
3. 頂級技術媒體（InfoQ、GOTO Conferences、ACM Digital Library）
4. 知名工程師部落格（Netflix Tech Blog、Uber Engineering、Cloudflare Blog）
5. 技術社群（Stack Overflow、Hacker News、Reddit r/programming）
6. 學術論文（arXiv、Google Scholar）

若無法找到足夠可信技術資料，標記「外部技術佐證不足」，不以推測填補。

### Step 5: 確立技術文章結構
依文章類型選擇敘事結構：

**問題解決型（預設）：**
1. 背景：這個問題為什麼重要？業界如何面對？
2. 挑戰：我們遇到了什麼具體困難？
3. 解法：我們的技術思路是什麼？
4. 實作：怎麼落地的？關鍵設計決策是什麼？
5. 結果：具體效益是什麼？數字是多少？
6. 延伸：下一步是什麼？讀者可以怎麼應用？

**架構分析型：**
1. 業務需求驅動：什麼業務需求決定了這個架構？
2. 架構全貌：整體設計說明
3. 關鍵設計決策：每個重要決策的 Why
4. 取捨分析：這個架構的優缺點
5. 演進路線：如何從現在走到未來

**技術選型型：**
1. 問題情境：需要解決什麼問題？
2. 候選方案：評估了哪些選項？
3. 評估維度：用哪些標準比較？
4. 最終決策：選了什麼？為什麼？
5. 上線後回顧：選對了嗎？

### Step 6: 撰寫技術文章
語言要求：
- **準確**：技術術語使用正確，不過度簡化導致失真
- **清楚**：每個概念第一次出現時附上簡短說明
- **有觀點**：說明「為什麼這樣做而非那樣做」，不只陳述事實
- **有佐證**：技術主張附上具體數字、程式碼片段、或外部參考
- **循序漸進**：鋪墊充足，讀者不需要先備知識也能跟上前半段

程式碼範例規範：
- 只放最關鍵的部分，不要貼完整原始碼
- 每個程式碼區塊都要有簡短說明
- 用 `// 說明` 或 `# 說明` 在關鍵行加上內聯注釋

### Step 7: 加入技術比較與外部佐證
整合外部研究，強化技術可信度：
- 與業界主流做法的對比表格
- 相關開源工具的比較（附官方連結）
- 引用的技術規範或論文（附 DOI 或 URL）
- 知名公司的類似實踐（附 Blog 連結）

每項外部資料都要附 URL 與日期。

### Step 8: 輸出技術文章
依 `輸出目標` 決定格式：
- `standalone`：輸出完整技術文章至 `project-insights/techblog-<project-slug>.md`
- `merge`（由 project-insight-synthesizer 觸發）：回傳結構化技術洞察區塊

### 整併輸出（merge，回傳給 project-insight-synthesizer）
回傳以下結構化內容：
- `tech_narrative`：核心技術突破敘事
- `architecture_block`：架構設計說明（若有）
- `benchmark_block`：效能數字與驗證結果（若有）
- `comparison_block`：與業界做法的比較（若有）
- `sources`：完整來源列表
- `confidence`：high / medium / low
- `notes`：資料缺口、高不確定性的說明

## 分層規則（Citation Rules）

### 專案事實層
- 標記格式：`來源：2026-04-15 專案記錄` 或 `來源：2026-04-15 逐字稿`
- 只能寫入專案 Markdown、逐字稿、或使用者直接確認的技術資訊
- 效能數字、架構決策、已驗證結果都屬於此層

### 外部技術資料層
- 標記格式：`外部技術資料：<標題> — <URL>（<日期>）`
- 適用於：業界做法比較、技術規範引用、工具文件、學術論文
- **任何不是來自專案本身的技術主張，都必須附外部來源，不可偽裝成原創發現**

### 資料不足時的處理
若無法找到足夠可信技術資料：
- 明確寫「外部技術佐證不足，以下為內部觀察，尚待與業界做法比對」
- 不以猜測填補技術細節

## 技術寫作品質要求
- 繁體中文為主，程式碼與技術術語保留英文
- 英文術語第一次出現時附中文說明（括號）
- 每個數字都要說明單位、測試環境與時間點
- 架構圖以文字清楚描述（元件 A → 元件 B 的流程）
- 避免：「突破性技術」、「業界領先」等沒有數據支撐的形容詞
- 鼓勵：「在 X 場景下，我們的方案比 Y 快了 Z%」這類有脈絡的具體比較

## 回報給使用者
完成後回報：
1. 文章主題與目標讀者
2. 核心技術突破點（2-3 點）
3. 外部技術佐證完整性（高 / 中 / 低）
4. 哪些技術主張有外部資料支持，哪些純來自專案事實
5. 建議補強的技術細節或資料缺口
6. 輸出檔案位置（standalone 模式）或整併狀態（merge 模式）
7. 建議的後續知識分享題材（基於本次分析）

## Integration with project-insight-synthesizer
若由 `project-insight-synthesizer` 觸發：
1. project-insight-synthesizer 傳入：專案名稱、專案 Markdown 摘要、技術主題、輸出目標 `merge`
2. 本 Skill 完成技術文章草稿後回傳結構化技術洞察區塊
3. project-insight-synthesizer 將其整併進專案 Markdown 的「技術亮點與對外發表素材」章節，保留來源標記
