# Project Insight Synthesizer

## Overview
將單次會議的資訊，轉成可持續累積的專案知識庫。每次執行都先讀取既有專案檔，再把新逐字稿與會議記錄中的事實、決策、風險、客戶動態與商業價值增量縫補進去，而不是重寫成一次性的摘要。

## Workflow

### Step 1: Collect source materials
優先取得以下資料，再開始整理：
1. 本次會議逐字稿（必需）
2. 本次會議記錄／摘要（強烈建議）
3. 現有專案 Markdown 檔（若已存在）
4. 同專案的其他相關補充文件（若使用者有提供）

若此 Skill 是接在 `meeting-transcription` 後面執行，直接使用其產出的：
- `*_逐字稿.md`
- `*_會議記錄.md` / `*_研究報告.md` / `*_學習筆記.md` / `*_整理筆記.md`

### Step 2: Detect projects and themes
先判斷這次會議涉及哪些專案、產品線、方案主題。

偵測原則：
- 明確專案名優先，例如 `DeviceOn`
- 同一會議若談到多個專案，分別更新各自檔案
- 若只談單一客戶需求、但未明說專案名，先依內容建立「候選主題」，不要硬猜正式專案名稱
- 若命名不確定，在檔案內保留「別名／待確認名稱」註記

### Step 3: Read the existing knowledge base before writing
這是 **stateful skill**，不可跳過。

每次更新前都必須：
1. 讀取既有專案 Markdown
2. 理解目前版本已知內容與結構
3. 比對本次會議有哪些內容屬於：
   - 新增事實
   - 舊資訊修正
   - 狀態推進
   - 風險升高／解除
   - 仍未確認的假設
4. 以「增量合併」方式更新，不要把先前仍有效的資訊刪掉

若檔案不存在，才建立新檔。

呼叫端（例如後端排程）通常只會列出已知專案名稱與檔名，不會附上檔案全文，避免把整份舊內容重複塞進輸入。一律用 Read 工具實際讀一次該專案的 Markdown 取得目前版本，這份讀檔結果也會是後續 Step 5 用 Edit 工具定點插入時所需的錨點依據。

> **⚠️ 不需要每次都讀完整份文件才能重新評估，只需重點讀取「專案目標」「欲解決的痛點」「目前進展」「技術亮點與重大突破」這幾個章節，即可判斷以下五個欄位。不得沿用舊值或憑印象直接複製上次結果，這五個欄位是本 Skill 的核心輸出：**
>
> 1. **目前階段**：對照最新事實，重新判斷專案所在開發／交付階段，確認是否有推進或退步
> 2. **核心價值主張**：確認目前最準確的客戶價值描述，若有新客戶訊號、技術突破或範疇調整須立即更新
> 3. **本次更新重點**：歸納這次會議帶來的最重要新資訊（不可只寫「無變化」，即使沒有大進展也應記錄確認事項）
> 4. **PM 判斷**：基於重點章節現況，重新給出 PM 視角的風險優先序、缺口與行動建議，不可沿用上次措辭
> 5. **成熟度百分比**：依本文件 Maturity scoring 章節的十階段標準逐項對照重新計算，不可直接沿用上次數值

### Step 4: Extract only evidence-backed updates
只寫入有來源依據的資訊。每條重要結論都要標記出處日期，必要時可同時標注：
- `逐字稿`：會議原話或直接推定事實
- `會議記錄`：整理後的決議、摘要、行動項目

嚴格區分：
- **已確認**：會議中明確說明或決議
- **高概率推定**：有多處訊號支持，但未被明講
- **待確認**：存在猜測成分、缺資料、或會議中仍未定案

對於不確定內容，使用 `待確認`、`推定`、`尚未定案` 等標記，不可寫成既成事實。


### Step 5: Update the project markdown by section
以主題式、可長期維護的結構更新，不要只按會議時間流水帳堆疊。優先維護以下章節：
- 專案定位與目標
- 欲解決的痛點
- 目前進展
- MP 狀態
- 客戶與目標市場
- 產業價值與對外敘事
- 技術亮點與重大突破
- 技術實踐與落地方式
- 技術細節與證據來源
- 風險／阻塞／待確認
- 下一步與建議
- 來源索引

若使用者明確表示這些 Markdown 未來會餵給 NotebookLM、簡報生成器或 PowerPoint 產生流程，務必把「可對外說的技術亮點」、「重大突破」、「實作證據」、「驗證方式」寫得更完整，方便後續直接抽成投影片。

可依專案特性增減，但以上欄位應盡量保留。

**現狀型章節（目前進展、MP 狀態、產業價值與對外敘事、技術亮點與重大突破、技術實踐與落地方式、技術現況與架構重點、風險／阻塞／待確認）必須遵守 `references/markdown-schema.md` 的「增量寫作鐵則」：逐筆累加、禁止整節覆寫、不可刪除舊 bullet（只能加註取代標記）。** 只有「會議更新紀錄」與「來源索引」本來就是按日期分段的日誌型章節，可以直接新增區塊。

#### 寫入方式：Edit 優先，Write 只用於建檔
為了避免每次更新都要把整份檔案內容當輸出重新生成一遍（檔案越大、輸出 token 越貴，也更容易撞到單次輸出長度上限），寫入方式分兩種情況：

- **檔案不存在（第一次建立這個專案）**：用 Write 工具寫入完整初始內容，套用 `markdown-schema.md` 的模板。
- **檔案已存在（這是絕大多數情況）**：一律用 Edit 工具做定點插入，不要用 Write 整份覆寫：
  1. 先用 Read 工具讀一次目前版本，確認各章節標題與插入點現有文字。
  2. 每個現狀型章節要新增的 bullet，以「該章節的下一個標題」作為錨點插入在它前面（例如 `old_string` 用 `\n## 技術實踐與落地方式`，`new_string` 用 `\n- 新 bullet ... \`來源：...\`\n\n## 技術實踐與落地方式`）；章節標題文字是 schema 規定的固定字串，可作穩定錯點。
  3. 「會議更新紀錄」新增本次的 `### YYYY-MM-DD` 區塊，同樣用 Edit 插入在既有區塊之前或之後，不要重寫整節。
  4. 若某個舊 bullet 需要加註取代標記（仍保留原文字），用 Edit 在該行原文字後面附加標記文字，不要整段替換。
  5. 純粹修正錯字、語句不通順但事實與日期來源不變的情況，才可以用 Edit 小範圍改寫單一 bullet 的文字。

只有在使用者明確要求「重新整理」、「精簡」、「重新排版」整份文件時，才整節甚至整份重寫；一般的會議增量更新不適用。

### Step 5.5: Self-check before saving
因為更新方式是「定點插入、不覆寫既有文字」，多數情況下舊內容不會被動到，不需要再做整份 diff 比對。寫入前確認：
1. 每個 Edit 操作的 `old_string` 是否唯一、確實對應到想插入的位置（不確定時先用 Read／Grep 確認，不要憑記憶猜測）。
2. 新插入的每個 bullet 是否都帶有日期來源標記。
3. 若這次操作是「整節重寫」（建檔，或使用者明確要求重整），才需要逐條比對舊版每個 bullet 是否仍存在（原樣或已標註取代），任何無故消失的視為誤刪並補回。
4. **`projects.json` 四個欄位是否都已覆寫為最新值**（不可遺漏任何一個）：
   - `"maturity"` 是否依新評分結果更新
   - `"maturityScore"` 是否依新評分結果重新計算
   - `"maturityLevel"` 是否與 `maturityScore` 同步
   - `"stage"` 是否依本次會議最新進度重新判斷（確認有沒有推進或退步）

### Step 6: Produce an executive update after writing
完成檔案更新後，回報使用者：
1. 這次更新了哪些專案
2. 每個專案目前大致進度
3. 主要風險或缺口
4. 初步判斷何時適合對外發表
5. 哪些判斷仍需更多資料支持
6. Step 5.5 自查結果：本次每個專案新增了幾條 bullet、標記了幾條取代、確認沒有無故消失的舊內容

## Output location and naming
預設把專案知識庫放在：

```text
<workspace>/project-insights/
  index.md
  index.html
  <project-slug>.md
  reviewer/
    index.html
    projects.json
```

建立或更新知識庫時，也要同步維護 reviewer：
- `project-insights/index.html`：根網址轉址頁，開啟 `/` 時要自動導向 `/reviewer/`
- `project-insights/reviewer/index.html`：前端 Markdown Reviewer
- `project-insights/reviewer/projects.json`：專案清單、階段、成熟度、最後更新日

`projects.json` 必須嚴格遵守以下格式：

```json
{
  "projects": [
    {
      "id": "project-slug",
      "slug": "project-slug",
      "name": "Project Display Name",
      "maturity": "not ready",
      "maturityScore": 10,
      "maturityLevel": "L2 問題初探",
      "stage": "版本迭代與平台補強中",
      "lastUpdated": "YYYY-MM-DD",
      "description": "一到三句話描述這個專案的範疇、核心技術領域與關鍵字，例如「與 Physical AI、Simulation、SO-101、GR00T 相關的機械手臂模擬訓練與控制平台」"
    }
  ]
}
```

`description` 欄位是給其他自動化流程（例如會議記錄分發、補充圖片分類建議）用來判斷「這段文字／這張圖片屬於哪個專案」的關鍵依據，**務必維護**：
- 建立新專案或第一次幫某專案產生 Markdown 時，依目前已知內容寫一個 1-3 句話的描述，包含領域、核心技術名詞、關鍵字（不是會議流水帳，是長期穩定的專案輪廓）
- 後續每次更新該專案時，若發現描述已經過時或不夠精確（例如專案範疇擴大、技術路線改變），直接覆寫成更準確的版本——這個欄位是「目前最佳描述」，不是日誌，不需要 append-only
- 若暫時無法判斷，先寫最保守、最不會誤導的描述，不要留空（留空會讓其他流程完全無法靠這個欄位做比對）

`maturity`、`maturityScore`、`maturityLevel` 三個欄位必須同時更新，保持一致：

| 十階段 | `maturityScore` | `maturityLevel` | `maturity`（大類，供 reviewer 篩選用） |
|--------|----------------|-----------------|---------------------------------------|
| L1 概念萌芽 | 0–9 | `"L1 概念萌芽"` | `"not ready"` |
| L2 問題初探 | 10–19 | `"L2 問題初探"` | `"not ready"` |
| L3 解法雛形 | 20–29 | `"L3 解法雛形"` | `"not ready"` |
| L4 技術驗證中 | 30–39 | `"L4 技術驗證中"` | `"internal only"` |
| L5 技術可行確認 | 40–49 | `"L5 技術可行確認"` | `"internal only"` |
| L6 客戶訊號初現 | 50–59 | `"L6 客戶訊號初現"` | `"soft launch ready"` |
| L7 試點進行中 | 60–69 | `"L7 試點進行中"` | `"soft launch ready"` |
| L8 試點驗證完成 | 70–79 | `"L8 試點驗證完成"` | `"public launch candidate"` |
| L9 上市準備中 | 80–89 | `"L9 上市準備中"` | `"public launch candidate"` |
| L10 公開上市候選 | 90–100 | `"L10 公開上市候選"` | `"public launch candidate"` |

- `maturity`：小寫英文大類值，供 reviewer 顏色標籤與篩選使用，只有以下四個合法值：`"not ready"` / `"internal only"` / `"soft launch ready"` / `"public launch candidate"`
- `maturityScore`：整數，取該次評分計算結果四捨五入後的百分比值（例如 `40`），每次更新後必須重新計算
- `maturityLevel`：字串，L 編號加中文名稱（例如 `"L5 技術可行確認"`），與 `maturityScore` 對應的十階段名稱

> **⚠️ 每次更新專案後，以下四個 `projects.json` 欄位都必須重新評估並覆寫，不可沿用舊值：**
> - `"maturity"`：依最新六面向評分結果，從對應表取得新大類值
> - `"maturityScore"`：依最新六面向評分結果重新計算百分比整數
> - `"maturityLevel"`：與 `maturityScore` 同步對應到正確 L 階段名稱
> - `"stage"`：依本次會議帶來的最新進度事實，重新判斷開發／交付階段文字，確認是否有推進或退步；不可因「感覺沒什麼大改變」就照抄舊值

`stage` 欄位為自由文字，描述專案目前所在的開發或交付階段，供 reviewer 快速辨識現況。建議使用以下參考值（可依實際情況調整措辭）：

| 參考值 | 適用情境 |
|---|---|
| `"概念驗證中"` | 尚在 PoC／可行性探索，尚無正式產品 |
| `"需求定義與規劃中"` | 需求收斂、功能範圍尚在拍板 |
| `"MVP 開發中"` | 首版功能正在實作，尚未可用 |
| `"MVP 試點中"` | 第一個可用版已交付，進行首次客戶驗證 |
| `"版本迭代與平台補強中"` | 功能逐步完善，正在處理穩定性、整合與周邊能力 |
| `"封閉測試與夥伴驗證中"` | 已有合作夥伴或種子客戶，做定向驗收或試用 |
| `"上市準備中"` | 功能已凍結，正在準備銷售材料、文件、定價與上市流程 |
| `"正式上線"` | 已對外可售或對外公告 |
| `"維護與持續優化"` | 主要功能穩定，以修補與小幅改善為主 |

選填原則：
- 依最近一次會議或里程碑的客觀描述來決定，不要超前假設
- 若專案同時有多條線（例如某功能仍在 PoC、另一功能已在試點），取主線狀態，並在 Markdown 內文說明細節
- 若無足夠資訊判斷，填 `"待確認"` 並在 Markdown 內文加 `待確認` 標記

若 reviewer 尚不存在，建立一個可直接預覽 Markdown 的靜態前端頁面；若已存在，增量更新專案列表與對應 metadata。根目錄若不存在 `index.html`，一併建立轉址頁，避免使用者打開 `/` 時只看到資料夾列表。

命名原則：
- 檔名用英文小寫與連字號，例如 `deviceon.md`
- 文件標題可保留正式名稱，例如 `# DeviceOn 專案洞察`
- 若無正式專案名，可先建立 `topic-<slug>.md`

## Required markdown structure
建立或更新專案檔時，優先使用 `references/markdown-schema.md` 的結構。

最少應包含以下章節：
- `# 專案名稱`
- `## 一頁摘要`
- `## 專案目標`
- `## 欲解決的痛點`
- `## 目前進展`
- `## MP 狀態`
- `## 客戶與目標市場`
- `## 產業價值與對外敘事`
- `## 技術現況與架構重點`
- `## 風險、阻塞與待確認`
- `## 下一步建議`
- `## 來源索引`
- `## 會議更新紀錄`

## Citation rules
技術細節、進度判斷、客戶訊息、商業主張都必須附來源。

### A. 會議內容來源
建議格式：
- `來源：2026-04-15 逐字稿`
- `來源：2026-04-15 會議記錄`
- `來源：2026-04-15 逐字稿；2026-04-15 會議記錄`

## Maturity scoring

> **⚠️ 每次執行不需要讀完整份專案 Markdown 才能評分，只需重點讀取「專案目標」「欲解決的痛點」「目前進展」「技術亮點與重大突破」這幾個章節，依照以下十階段標準逐項對照，重新計算成熟度百分比。不可沿用上次數值，不可憑印象估算。**

每次更新後，以下五個欄位須依最新評估結果重新填寫於專案 Markdown 的一頁摘要區：

| 欄位 | 說明 |
|------|------|
| **目前階段** | 當前開發／交付階段文字（見 stage 欄位參考值） |
| **核心價值主張** | 目前最準確的一到兩句客戶價值描述 |
| **本次更新重點** | 這次會議帶來的最重要新資訊（必填，不得留空） |
| **PM 判斷** | PM 視角的風險優先序、缺口與建議行動 |
| **成熟度** | 百分比 + 階段名稱，例如 `40%（L5 技術可行確認）` |

---

### 十階段成熟度定義

| 階段 | 百分比 | 名稱 | 一句話描述 | 對應大類 |
|------|--------|------|-----------|----------|
| L1 | 0–9% | 概念萌芽 | 僅有初步想法，問題、解法、客戶均尚未定義 | Not Ready |
| L2 | 10–19% | 問題初探 | 痛點有初步描述，但市場規模或客戶真實性未驗證 | Not Ready |
| L3 | 20–29% | 解法雛形 | 有初步解法構想，但可行性未確認，無技術證據 | Not Ready |
| L4 | 30–39% | 技術驗證中 | 有 PoC 或技術驗證進行中，但結果尚未明確 | Internal Only |
| L5 | 40–49% | 技術可行確認 | PoC 完成且有正向結果，內部共識形成，尚無客戶接觸 | Internal Only |
| L6 | 50–59% | 客戶訊號初現 | 已有客戶接觸、意向確認或試點規劃，但尚未交付 | Soft Launch Ready |
| L7 | 60–69% | 試點進行中 | 有實際試點或 MVP 交付進行中，有初步使用回饋 | Soft Launch Ready |
| L8 | 70–79% | 試點驗證完成 | 試點有正向結果，有可引用案例或量化數據 | Public Launch Candidate |
| L9 | 80–89% | 上市準備中 | 功能趨於完整，對外敘事清楚，銷售材料與定價規劃中 | Public Launch Candidate |
| L10 | 90–100% | 公開上市候選 | 具備完整價值主張、案例與對外材料，可全面對外發表 | Public Launch Candidate |

### 評分標準（六個面向，每次必須逐項對照）

每次評分前必須逐項確認以下六個面向：

| 面向 | 評估問題 | 0分（完全未達成） | 8分（部分達成） | 16分（完全達成） |
|------|---------|-----------------|----------------|----------------|
| **① 問題定義清晰度** | 痛點是否明確、可驗證、有佐證？ | 問題模糊或尚未描述 | 有初步痛點描述，但缺佐證 | 痛點清楚、有市場或客戶佐證 |
| **② 解法成形程度** | 技術路線是否確定、功能是否可展示？ | 僅有想法，無具體方案 | 有方案構想，部分技術確認 | 功能可展示，技術路線明確 |
| **③ 客戶／PoC／試點訊號** | 是否有真實客戶接觸、意向或試點結果？ | 無任何客戶接觸 | 有初步接觸或意向，但未試點 | 有試點、PoC 或正式客戶回饋 |
| **④ 里程碑進度（MP 狀態）** | 是否已達成或接近 Milestone Product？ | 距 MP 尚遠 | 接近 MP 或部分里程碑達成 | 已有 MP 或超越 MP |
| **⑤ 對外敘事完整度** | 價值主張、差異化、市場定位是否清楚？ | 無法清楚陳述對外說法 | 有初步說法，但不夠完整 | 對外敘事完整，可直接使用 |
| **⑥ 技術與交付風險可控性** | 已知風險是否有對策、阻塞是否可解？ | 有重大未解風險或阻塞 | 部分風險有對策，仍有不確定項 | 主要風險已有對策，交付可控 |

**計分方式：**
- 六個面向各給 0、8、16 分，合計最高 96 分，換算為百分比（÷96×100，四捨五入到最近 10%）
- 再依百分比對應上方十階段表中最接近的階段
- 不要把這個分數包裝成事實；它是 PM 視角的工作判斷

## Style requirements
文件必須同時滿足：
- **高度可讀性**：一頁摘要清楚，標題分層穩定
- **專業性**：區分事實、推定、風險、建議
- **產業宣傳性**：在不誇大的前提下，把市場價值、差異化、客戶場景講清楚
- **可追溯性**：技術與關鍵商業敘事有日期來源可回查
- **簡報友善性**：技術亮點、重大突破、驗證結果與可對外說版本要能被後續工具直接抽成 slide
- **來源分層清楚**：會議原文 vs 外部 insights 必須肉眼可辨，不可混淆

寫作原則：
- 中文一律用繁體中文
- 英文專有名詞保留英文
- 不使用空泛形容詞堆砌內容
- 若會議證據不足，不要補腦補滿
- 若發現資訊互相矛盾，明確標示矛盾點與待確認項目

## Scripts

### scripts/extract_supplementary_images.py
- **Purpose:** 從會議補充文件（`.pptx`/`.xlsx`/`.pdf`）中取得可供分類判斷的內容圖片，並輸出 `manifest.json` 供後續判斷哪張圖該插入哪個專案文件。
  - **PPTX：整頁截圖**，不是擷取內嵌圖片物件。透過 `unoconvert`（unoserver 的輕量 client）把每個 PPTX 丟給遠端的 `clawpm-unoserver` sidecar 容器轉成 PDF，再用 pymupdf 把每一頁渲染成一張 PNG（約 144 DPI）。這樣每張投影片的版面、文字、圖表都完整呈現，而不會只抓到裝飾性的小圖示/Logo（這些圖示原本散落在投影片裡，跟專案內容無關，對分類判斷沒有幫助）。
  - **XLSX/PDF：仍是擷取內嵌圖片物件**（過濾掉過小的圖示／Logo），維持原邏輯不變。
- **不屬於本 skill 主工作流程**：本腳本是給後端「記錄分發（Step 6）」流程呼叫的，用來把補充文件中的圖片插入各專案的 `record-{slug}/{date}.md`，而不是插入本 skill 維護的專案知識庫 Markdown。
- **Dependencies:** `python3 -m pip install python-pptx openpyxl pymupdf unoserver`。
  - 真正跑 LibreOffice 的是獨立的 `clawpm-unoserver` 容器（見 `docker/converter/Dockerfile`，由 `docker-compose.yml` 啟動），OpenClaw 主容器裡的 `unoserver` 套件只提供 `unoconvert` 這個 client 指令，不需要也沒裝 LibreOffice——把體積較大的 LibreOffice + 字型留在 sidecar，主 image 維持瘦身。
  - 連線位置由環境變數 `UNOSERVER_HOST`（預設 `host.docker.internal`）/ `UNOSERVER_PORT`（預設 `2003`）決定，由 `backend/src/containers/ContainerManager.js` 在建立 per-user 容器時注入。若 `clawpm-unoserver` 沒在跑，PPTX 轉換會直接報錯並提示確認該容器狀態。
- **Arguments:**
  - `<inputs>` – 一個或多個補充文件路徑（`.pptx`/`.xlsx`/`.pdf`，其他格式會被跳過並記錄在 manifest 的 `skipped`）
  - `--output-dir`（必填）– 圖片輸出目錄
  - `--manifest` – manifest JSON 路徑（預設：`<output-dir>/manifest.json`）
  - `--min-size` – 最小寬高（像素），用於過濾圖示／Logo（預設：120；PPTX 整頁截圖通常遠大於此值，此參數對 PPTX 基本不生效）
  - `--max-images` – 單次執行最多萃取張數（預設：200）
- **Outputs:** 圖片檔案 + `manifest.json`（每張圖片含 `file`、`sourceFile`、`sourceType`、`location`、`width`、`height`、`captionHint`）。**注意：`captionHint` 不是圖片內容的描述**，只是該投影片第一個文字框的文字（PPTX）或該頁文件第一行文字（PDF），常常跟圖片實際畫面無關，只能當文件結構上的弱線索，呼叫端（Agent）應實際用 Read 工具檢視圖片內容，自行判斷與哪個專案相關，不可只憑檔名或 captionHint 猜測。
- **若使用者用的是 custom/Azure OpenAI-compatible provider**：請確認 `openclaw.json` 裡該模型的 `models[].input` 陣列有包含圖片能力宣告（不能只有 `["text"]`），否則 OpenClaw 不會把圖片真正的可視內容傳給模型，Read 工具對圖片只會回傳「檔案存在」之類的訊息。
