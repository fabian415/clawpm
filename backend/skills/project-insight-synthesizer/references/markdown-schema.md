# Project Insight Markdown Schema

以下模板用於每個專案的主題式知識庫文件。目標不是產出一次性的會議摘要，而是持續更新、可被 PM 長期維護與閱讀的專案洞察頁。

## 增量寫作鐵則（Append-only Rules）

以下章節屬於「現狀型章節」，必須以**逐筆累加**方式維護，禁止整節覆寫：
`目前進展`、`MP 狀態`、`產業價值與對外敘事`、`技術亮點與重大突破`、`技術實踐與落地方式`、`技術現況與架構重點`、`風險、阻塞與待確認`。

規則：
1. **新事實＝新增 bullet**，永遠加在該小節最下方，不可把舊 bullet 的文字改寫成新內容。
2. **不可刪除舊 bullet**。即使該事實已經過期或被取代，也要保留原文字，只在後面加註取代標記：
   `(已於 YYYY-MM-DD 由「{{新事實摘要}}」取代/延伸，詳見下方 YYYY-MM-DD 條目)`
3. 同一件事如果在多次會議中被重複確認（例如某風險持續存在），不要合併成一條沒有日期的敘述，而是讓最新確認的 bullet 補一個來源日期，舊 bullet 仍保留。
4. 唯一允許「修改既有 bullet 文字」的情況：純文字潤飾、事實與日期來源完全不變（例如修正錯字、調整語句通順度）。
5. 每次更新前，先列出舊版本該章節的全部 bullet，逐條確認新版本是否仍存在（原樣或已標註取代）；若有消失、且無取代標記，視為誤刪，必須補回。

> 範例（技術亮點與重大突破章節）：
> ```
> - Sim2Real 路徑完成首次重大突破，誤差收斂至可用範圍 `來源：2026-04-24 會議記錄`
> - Simulation backend 已能支援多步驟 service、進度查詢與推斷 `來源：2026-06-12 BiWeeklyReport.xlsx`
> ```
> 而不是只保留最新一條、把 04-24 的 Sim2Real 突破整句刪掉。

## Writing Notes
- 若同一專案跨多次會議，依照上方「增量寫作鐵則」逐筆累加主題欄位的內容，不可整節改寫成單一最新狀態；本次變化同時也要寫進「會議更新紀錄」。
- 若資訊互相矛盾，不要刪掉舊資訊；在對應段落加上「版本差異／待確認」。
- 若客戶名稱敏感，可寫成客群代稱，但內部版建議保留真實名稱。若要對外版，另外輸出裁切版本。
- 寫完整份檔案前，務必完成「更新前差異自查」：逐一比對舊版每個現狀型章節的 bullet 是否仍出現在新版中（原樣或已標註取代），若有無故消失，先補回再輸出。
