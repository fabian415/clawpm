# Market Insight Synthesizer

## Overview
從行銷與市場推廣角度出發，結合專案事實與外部市場調查，產出一份可直接交給客戶、業務、或用於公開推廣的 Markdown 文件。

**核心原則：**
- 語言親切易懂，初學者也能看懂
- 聚焦在「這對我有什麼好處」，而非技術細節
- 所有市場數字與外部主張必須附可驗證來源
- 嚴格區分專案事實層與外部市場洞察層

若由 `project-insight-synthesizer` 觸發，本 Skill 產出的行銷區塊將整併進既有專案 Markdown；若為獨立呼叫，輸出獨立的行銷文件。

## 輸入

| 欄位 | 必要性 | 說明 |
|------|--------|------|
| 產品／方案名稱 | 必要 | 行銷對象 |
| 產品基本說明 | 必要 | 功能、用途、目標客群的初步描述 |
| 專案 Markdown 或會議摘要 | 選擇性 | 由 project-insight-synthesizer 傳入；作為行銷素材的事實基礎 |
| 目標讀者 | 選擇性 | 例如：IT 採購主管、中小企業老闆、工廠廠長；未指定則預設為非技術決策者 |
| 行銷重點 | 選擇性 | 可指定：痛點解決、ROI 佐證、競品比較、客戶案例、趨勢搭車；未指定則執行完整行銷文件 |
| 輸出目標 | 選擇性 | `standalone`（獨立行銷文件）或 `merge`（整併進既有專案 Markdown） |

## Workflow

### Step 1: 確認目標讀者與行銷角度
依使用者輸入，確認這份文件要給誰看、要說服他們什麼：
- **目標讀者**：技術人員 / 業務 / 採購主管 / 最終用戶 / 一般大眾
- **行銷目標**：認識產品 / 產生興趣 / 評估採購 / 建立信任
- **語氣設定**：親切說明型（預設）/ 商業說服型 / 技術信任型

若使用者未指定，預設為「面向非技術決策者，語氣親切，強調業務價值」。

### Step 2: 提取並隔離專案事實
若有傳入專案 Markdown 或會議摘要，先提取以下內容作為**事實基礎**，標記來源日期：
- 產品功能與定位
- 已解決的客戶問題
- 已有的客戶案例或 PoC
- 已知的效益數字（速度、成本節省等）
- 當前進展與成熟度

這些是可直接轉化為行銷敘述的事實。**不可在此步驟混入外部推論。**

### Step 3: 執行外部市場研究
針對以下面向進行外部研究，為行銷文件提供市場佐證：

搜尋優先順序：
1. 市場調研機構（Gartner、IDC、Forrester、Statista）
2. 正規科技 / 財經媒體（TechCrunch、Forbes、MIT Technology Review）
3. 產業協會報告
4. 競品官方行銷頁面（了解市場說法，非技術細節）
5. 客戶案例與用戶評價平台

研究面向：
- **市場規模與成長**：這個問題有多大？市場有多少錢？
- **客戶痛點現況**：業界普遍面對什麼問題？
- **解決方案效益佐證**：類似方案幫助客戶達成什麼成效？
- **市場趨勢**：為什麼現在是對的時機？

若找不到足夠資料，標記「外部佐證不足」，不以推測填補。

### Step 4: 選定行銷敘事框架
依目標讀者選擇主要敘事框架：

- **痛點導向**：先說問題，再說解法，最後說效益
- **趨勢搭車**：先說市場趨勢，再說我們的定位，再說為何現在要做
- **成果導向**：先說客戶得到了什麼，再說怎麼做到的
- **比較導向**：先說現有方案的不足，再說我們如何不同

預設使用「痛點導向」框架，因為最容易引起共鳴。

### Step 5: 撰寫行銷內容
依框架撰寫，語言要求：
- **不用術語**：把技術詞彙翻譯成業務語言（例如：「即時監控」→「問題發生前就能收到通知」）
- **數字具體化**：「大幅提升效率」→「平均縮短 30% 的停機時間」（附來源）
- **場景化描述**：「可以想像一下……」引導讀者代入情境
- **主動語態**：「您的團隊可以……」而非「系統能夠……」
- **每段不超過 5 行**：避免大段文字

### Step 6: 整理社會佐證與市場數字
收集可信的市場數字與佐證，強化說服力：
- 市場規模數字（附來源與年份）
- 客戶案例或成效數字（附來源）
- 產業採用趨勢（附來源）
- 第三方評分或認可（若有）

每項數字都要附來源 URL 與日期。

### Step 7: 輸出行銷文件
依 `輸出目標` 決定格式：
- `standalone`：輸出完整行銷 Markdown 至 `project-insights/marketing-<project-slug>.md`
- `merge`（由 project-insight-synthesizer 觸發）：回傳結構化行銷區塊

## 輸出格式

### 獨立行銷文件（standalone）
輸出至 `project-insights/marketing-<project-slug>.md`：

```markdown
# <產品名稱>：<一句話價值主張>

> 整理日期：YYYY-MM-DD
> 本文結合產品事實與外部市場研究，為推廣用途而整理，非技術規格文件。

## 一句話說清楚
[用一句話說明這個產品解決什麼問題、對誰有價值]

## 您是否也有這樣的困擾？
[2-4 個具體客戶痛點，用第二人稱，讓目標讀者能代入]
- 痛點 1：[場景化描述]
- 痛點 2：[場景化描述]
- 痛點 3：[場景化描述]

> 市場現況：[相關市場數字佐證]
> 外部資料來源：<標題> — <URL>（<日期>）

## 我們的解法
[用平易近人的語言說明產品如何解決上述痛點，不要放技術架構圖]

### 核心功能（用客戶語言描述）
| 功能 | 對您的意義 |
|------|-----------|
| [功能名] | [帶來的業務好處] |

## 實際效益
[具體數字或客戶回饋]
- ✓ [效益 1]（來源：[日期] 專案記錄 / 外部資料來源：<URL>）
- ✓ [效益 2]
- ✓ [效益 3]

## 為什麼現在？市場正在發生什麼
[說明產業趨勢，讓客戶感受到「現在不做就落後了」的緊迫感]

> 外部資料來源：<標題> — <URL>（<日期>）

## 適合哪些人
- **最適合**：[目標客群 1]
- **也適合**：[目標客群 2]
- **暫不適合**：[排除的場景，若有的話]

## 客戶怎麼說（若有案例）
> "客戶引言" — 客戶名稱／職稱（來源：[日期] 專案記錄）

## 常見問題
**Q：[常見疑問 1]**
A：[平易近人的回答]

**Q：[常見疑問 2]**
A：[平易近人的回答]

## 下一步
[明確的行動呼籲，例如：申請試用、聯繫業務、下載白皮書]

---

## 資料透明度說明
- 產品事實來源：[日期] 專案記錄
- 外部市場數字完整性：高 / 中 / 低
- 主要不確定性：[說明]

## 來源索引
| # | 標題 | URL | 日期 |
|---|------|-----|------|
```

### 整併輸出（merge，回傳給 project-insight-synthesizer）
回傳以下結構化內容：
- `marketing_narrative`：一頁行銷敘事
- `pain_points`：客戶痛點列表與佐證
- `benefits_block`：效益數字與來源
- `market_evidence`：市場佐證數字
- `sources`：完整來源列表
- `confidence`：high / medium / low
- `notes`：資料缺口說明

## 分層規則（Citation Rules）

### 專案事實層
- 標記格式：`來源：2026-04-15 專案記錄` 或 `來源：2026-04-15 逐字稿`
- 只能寫入專案 Markdown、逐字稿、或使用者直接確認的資訊

### 外部市場洞察層
- 標記格式：`外部資料來源：<標題> — <URL>（<日期>）`
- 適用於：市場規模數字、產業趨勢、競品說法、第三方效益佐證
- **任何不是來自專案事實的描述，都必須附外部來源，不可憑空推測**

### 資料不足時的處理
若無法取得足夠可信資料：
- 明確寫「市場佐證資料不足，以下為現有資訊的局部說明」
- 不以無來源的推測填補說服力

## 語言品質要求
- 繁體中文為主，英文專有名詞保留英文
- 每個技術術語都要附上白話解釋（括號或換句話說）
- 避免：「革命性」、「顛覆性」、「業界唯一」等空泛形容詞，除非有數字佐證
- 善用：「想像一下……」、「您的團隊可以……」、「就像……一樣」等帶入感語句
- 數字優先：能用數字說的不用形容詞

## 回報給使用者
完成後回報：
1. 執行了哪些行銷面向分析
2. 主要行銷訴求（2-3 點）
3. 外部佐證完整性（高 / 中 / 低）
4. 哪些主張有外部來源支持，哪些僅來自專案事實
5. 建議補強的資料缺口
6. 輸出檔案位置（standalone 模式）或整併狀態（merge 模式）

## Integration with project-insight-synthesizer
若由 `project-insight-synthesizer` 觸發：
1. project-insight-synthesizer 傳入：專案名稱、專案 Markdown 摘要、目標讀者、輸出目標 `merge`
2. 本 Skill 完成行銷文件後回傳結構化行銷區塊
3. project-insight-synthesizer 將其整併進專案 Markdown 的「行銷素材」章節，保留來源標記
