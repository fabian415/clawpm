# Skill Creator

## Overview
把使用者用白話描述的需求，轉成一份可直接放進 `skills/<slug>/SKILL.md` 的技能定義草稿。輸出**只是草稿**，一律寫到指定的暫存草稿檔，不可直接建立或覆寫 `skills/` 底下任何正式技能資料夾——是否採用、何時正式建立，由使用者在 ClawPM 網頁上的技能管理頁決定。

## 輸入
呼叫方（`/api/skills/generate`）會在 prompt 中提供：
- 使用者對這個新技能的需求描述（自然語言，可能簡短、不完整）
- 草稿輸出路徑（例如 `workspace/.skill-drafts/<draftId>.md`）

## Workflow

### Step 1: 釐清技能的核心任務
從需求描述中判斷：
1. 這個技能主要「輸入」什麼（會議逐字稿？專案文件？使用者上傳的檔案？其他技能的輸出？）
2. 主要「輸出」什麼格式與去向（一份 Markdown 報告？一組結構化資料回傳給別的技能？寫入 `project-insights/`？）
3. 什麼情境下應該被觸發（使用者會說什麼樣的話、或哪個既有技能執行完後會接著呼叫它）

若需求太模糊以致無法產出一個可執行的 Workflow，仍要產出草稿，但在草稿最上方以 HTML 註解 `<!-- 待確認：... -->` 列出需要使用者補充的關鍵問題，不要求使用者先回答完才產出。

### Step 2: 撰寫 frontmatter
遵守本 workspace 既有技能一致的格式（可參照 `skills/swot-analyzer/SKILL.md`、`skills/tech-analyzer/SKILL.md`）：

```
---
name: <slug，小寫英數字與連字號，簡潔描述功能>
description: <一段話，必須同時包含：這個技能做什麼、適用情境（使用者可能會說的觸發語句，至少 2-3 個例句）、若與其他技能有整合關係也一併說明>
---
```

`description` 是唯一影響「AI 何時會自動選用這個技能」的欄位，務必寫得具體、可比對，不要只寫功能摘要而漏了觸發語句。

### Step 3: 撰寫 Workflow
以有編號的 `### Step N: ...` 小節列出具體步驟，每一步要說清楚「做什麼」與「輸出到哪裡／回傳什麼」，比照既有技能的分步風格。若技能需要輸出檔案，明確給出檔案路徑慣例（例如 `project-insights/<自訂前綴>-<project-slug>/<timestamp>.md`）。

若這個技能是要處理某專案的既有資料（會議記錄、專案 Markdown、上傳文件），提醒使用者：正式執行時呼叫方會告知實際的專案 slug 與檔案路徑，草稿裡用 `<project-slug>` 之類的佔位符即可，不用寫死。

### Step 4: 撰寫輸出格式與回報段落
比照既有技能慣例，加上「## 輸出格式」段落（若有固定輸出檔案格式）與「## 回報給使用者」段落（列出完成後要跟使用者回報哪些重點）。

### Step 5: 寫入草稿檔
把完整的 SKILL.md 內容（含 frontmatter）寫入呼叫方指定的草稿路徑。**只寫這一個檔案**，不要在 `skills/` 底下建立任何新資料夾或檔案。

完成後，用一到兩句話跟使用者總結這份草稿的核心設計（技能要做什麼、什麼時候會被用到），並提醒需要透過技能管理頁確認後才會正式建立為可用技能。

## 注意事項
- 不可覆寫或修改任何既有技能（`skills/` 底下其他資料夾），只能寫草稿檔
- 草稿內容必須是合法、完整的 SKILL.md（含 `---` frontmatter 區塊），前端會直接把這份內容載入 Markdown 編輯器
- 若使用者需求明顯與既有技能高度重疊（例如又要一個「做 SWOT」的技能），在草稿註解中提醒使用者可考慮改用「複製既有技能」而非全新建立，但仍照常產出草稿
