# outline.json Schema（Mode B 產出 — 故事大綱，需使用者確認）

**這是在動手做 PPTX 之前的必經關卡。** 你不是先想版面,是先把「這份簡報為什麼存在、
每一頁憑什麼存在」想清楚,寫成大綱,讓使用者讀過、確認「這故事夠不夠打動人心」之後,
才進入 Mode C 真正產出 spec 與 pptx。使用者也可能不滿意,回饋意見要求重新設計故事線
——這時你要重新走一次 Mode B(帶著使用者的回饋),而不是硬做同一版。

## 核心原則:先寫目的,再設計內容

**每次設計大綱,無論是整份的 deck_purpose,還是每一頁的 purpose,都必須先把「完整的目的」
寫清楚,再往下寫這一頁要放什麼。** 目的沒想清楚就不要寫內容——這是避免整份簡報變成
「流水帳」(把資訊平鋪直敘倒出來,頁與頁之間沒有因果、沒有推進)的唯一辦法。

對每一頁自問並寫下(這就是 `purpose` 欄位要回答的):
1. **這頁為什麼必須存在?**——拿掉這頁,故事還完整嗎?如果拿掉沒差,這頁就是流水帳,刪掉或合併。
2. **它在故事線裡做什麼工作?**——是拋出鉤子、揭露痛點、給出轉折洞察、提出解法、佐證、算效益,還是催促行動?
3. **它從哪裡接過來、要帶去哪裡?**——上一頁留下的問題/情緒,這頁怎麼接;這頁又留下什麼讓下一頁接。

沒有明確目的的頁面(例如「反正這個資訊還沒講,放一頁條列」)一律不要放進大綱。

## 規則

- `pages` 陣列**只放最終確定的頁數**,順序就是簡報順序。不要先塞一堆再說,每一頁都要先通過
  「拿掉會不會少了什麼」的檢驗。
- `purpose` 一律用**完整句子**寫清楚目的,禁止只寫「介紹背景」「補充資訊」這種空泛字眼。
- `connects_from` / `connects_to` 逼你檢查頁與頁之間有沒有因果關係,不是各自獨立的資訊塊。
- `self_check` 要誠實。如果有頁面目的薄弱、與其他頁重複、或故事斷裂,寫出來並回去修正
  大綱本身,直到都是 true 才輸出給使用者確認。
- 這是**唯一產出**;Mode B 不寫 deck spec、不呼叫 deck_builder。
- 若這次是依使用者回饋重新設計(`based_on_feedback` 有值),要明確回應回饋裡點出的問題
  (例如「上一版太像流水帳」「這句話沒有打中受眾」),不是換個詞敷衍過去。
