# Storytelling — 故事線、版面變化、圖片語意配對

目標:讓簡報**有故事、不生硬、每次不一樣、圖片不硬塞、不是流水帳**。

## 0. 先寫目的,再設計內容(最重要的一條)

**不管是整份簡報的大綱,還是單一頁的內容,動筆前都要先把「完整的目的」寫清楚。**
目的沒想清楚,不准往下寫內容。這是唯一能避免「流水帳」的辦法——流水帳的本質就是
內容平鋪直敘倒出來,卻沒人問過「這段憑什麼在這裡」。

對整份簡報:先寫 `deck_purpose`——對誰、要改變什麼認知或促成什麼行動、為什麼是現在。
不是「介紹專案」這種空話。

對每一頁:先寫 `purpose`——回答三個問題再動筆:
1. 這頁為什麼必須存在?拿掉它,故事還完整嗎?不完整才留。
2. 它在故事線裡做什麼工作(拋鉤子/揭痛點/給洞察/提解法/佐證/算效益/催行動)?
3. 它接住了上一頁的什麼,又要丟給下一頁什麼?

這個「先寫目的」的流程要在 **Mode B(大綱)** 完成並讓使用者確認,通過後才進 Mode C
把大綱轉成 deck spec。細節格式見 `references/outline-schema.md`。

## 1. 一條故事線串全場

不要把「章節」機械式倒出來。先想清楚一條線,每頁只推進一步:

```
鉤子(為何現在該聽)→ 痛點(觀眾/客戶的真實困擾)→ 洞察(關鍵轉折/我們看到什麼)
→ 解法(我們怎麼解)→ 佐證(數據/Demo/架構)→ 效益(量化價值)→ 行動(下一步/邀請)→ 來源
```

依受眾調整重心:
- **高層**:痛點—效益—行動要重,技術點到為止;多用 `big_number`、`quote`、`summary`。
- **技術評審**:解法—架構—可行性要厚;多用 `process`、`image_full`、`two_column`。
- **潛在客戶**:痛點—解法—成果—效益;多用 `hero_image`、`stats`、`image_grid`。

**先寫「一頁一句話」的大綱,自己讀一遍確認流暢,再寫 spec。** 每頁自問:這頁的一個訊息是什麼?它有沒有承接上一頁、帶出下一頁?

## 2. 避免生硬與流水帳

- 標題用**短句或提問**帶出觀點(「提案為什麼總是慢?」),別只放名詞(「痛點」)。
- 一頁一個核心訊息;細節放內文,不要把長段落貼上去。
- 資訊有起承轉合:痛點頁要讓人有感,解法頁要對應那個痛點,效益頁要回扣核心訊息。
- 用 `quote` 放一句真實客戶原話或核心主張,做情緒轉折。
- **警覺「流水帳訊號」**:如果一頁的內容是「反正這個資訊還沒講,放一頁條列」,或某頁跟
  前後頁沒有因果關係、拿掉也不影響理解,這就是流水帳,大綱階段要直接刪除或合併。
- **條列不是預設值**。條列頁(尤其純文字 `bullets`)是故事線裡「需要並列陳述幾個同等重要
  的點」時才用的工具,不是每頁的預設骨架。同一份簡報條列頁佔比過高,就是流水帳的徵兆——
  換成 `quote`/`big_number`/`process`/`two_column` 等更有畫面感的頁型來承載同樣的內容。
- 每頁的 bullets 數量寧少勿多(2–4 點),每點都要能通過「這點不寫,故事會少什麼」的檢驗。

## 3. 每次版面都不同(但不脫離模板)

- 設 `meta.style`(使用者選)+ `meta.layoutSeed`(後端給的整數),讓配色與版面預設每次不同。
- **主動變換版面變體**(見 deck-spec-schema「版面變體」):
  - 封面試 `variant: bold / hero`,不要每次都 brand。
  - 條列頁試 `variant: cards`,或 `imageSide: left` 把圖放左邊。
  - 數據頁試 `variant: grid`(2×2);對比頁試 `variant: divider`。
- 混搭頁型,不要整份都 bullets。把「重點對比」做成 `two_column`,「關鍵數據」做成 `big_number` 或 `stats`,「流程」做成 `process`,「主視覺」做成 `hero_image`。
- 換開場與故事切法(從痛點切入 / 從一個數據切入 / 從一句話切入)。
- 目標:同一專案生兩次,**結構、版面、配色、頁型組合都明顯不同**。

## 4. 圖片語意配對(不硬塞)

先讀懂每張 `_assets` 圖在講什麼(discovery 的 image_inventory),再決定放不放:

| 圖片性質 | 放哪種頁 | 注意 |
|---|---|---|
| 架構圖 / 流程示意 | `image_full`(完整不裁切)或 `process` | 圖說寫「這張圖說明的流程/關係」 |
| 真實照片 / 主視覺 | `hero_image`(裁切填滿,畫面感強) | 不要用在需要看細節的圖表 |
| 多張成果截圖 | `image_grid` | 每張給一句 caption |
| 單一重點畫面 | `bullets` 右側配圖 或 `image_full` | 圖要支撐該頁論點 |

鐵則:
- **配不上語意就不放**。寧可純文字/數據卡,也不要塞一張不相干的圖。
- 圖說(caption)要說明「這張圖證明/展示什麼」,不是只寫檔名或空話。
- 一張圖只用一次;不要同一張到處重複。
- 圖表細節多的用 `image_full`(不裁切);照片主視覺才用 `hero_image`(會裁切填滿)。

## 5. 引用與可信度

- 外部數字/競品/趨勢,句末附「（來源：<標題> — <URL>）」。
- 倒數第二頁放 `sources`,把用到的外部來源(標題 + URL)列出。
- 沒有可信來源的數字不要寫;寧可標「資料有限」。
